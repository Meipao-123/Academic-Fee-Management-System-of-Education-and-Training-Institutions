### 1.2 影响分析

| **SRS 完整性** | 致命 | 无法作为 IEEE 830 合规的 SRS 交付 |
| **架构设计输入** | 致命 | 缺少 NFR（性能/安全）将导致后续 ADR/ASD 设计无约束基线 |

### 1.3 根因分析

使用 **5-Why** 法追溯根因：

| Why# | 问题 | 回答 |
|---|---|---|
| 1 | 为什么 SRS 在第 422 行截断？ | A4 智能体生成 SRS 时，LLM 上下文窗口或输出 token 限制导致内容被截断 |
| 2 | 为什么输出 token 不够？ | SRS 目标 ≥15000 字，单次生成可能超出模型的 max_tokens 设置 |
| 3 | 为什么没有检测到截断？ | A4 生成后缺乏自动完整性校验（如章节覆盖率检查、目录与实际内容比对） |
| 4 | 为什么 A4 没有分块生成？ | A4 智能体设计时未考虑超长文档的分段生成 + 拼接策略 |
| 5 | **根因** | **需求工程流水线中缺少"文档完整性自动校验"环节——A4 输出后应自动检查目录承诺章节是否全部存在** |

### 1.4 修复措施

1. 从 `raw/notes/`、`需求问题清单-v1.0.md`、UML 模型中提取素材
2. 补全 §3.1.5 RPT 模块（FR-RPT-002 完成 + FR-RPT-003/004/005 新增）
3. 新增 §3.1.6 ADM 模块（FR-ADM-001~005）
4. 新增 §3.2 外部接口需求（5 条 IF-SW + 用户/硬件/通信接口）
5. 新增 §3.3 数据需求（10 实体逻辑模型 + 6 组数据字典枚举）
6. 新增 §3.4 非功能需求（15 条 NFR）
7. 新增 §4.1 TBD 清单（24 项）
8. 新增 §4.2 RTM 溯源矩阵（29 条 FR 全覆盖）
9. 新增 §4.3 变更记录

### 1.5 修复验证

| 验证项 | 方法 | 结果 |
|---|---|---|
| 文件行数 | `wc -l` | 421 → 771 行 (+83%) |
| 目录章节覆盖率 | grep 标题 vs 目录 | 17/17 (100%) |
| compile.js | `node compile.js` | 0 错误 0 警告 |
| A5 重验证 | 验证报告 v2.1 | 维度⓪ 通过，INT-001 关闭 |

### 1.6 经验教训

1. **AI 生成超长文档必须分块**：单次 LLM 调用的 token 限制是已知约束，SRS 这种万级字文档应采用"骨架→逐章填充→拼接"策略
2. **生成后自动校验不可或缺**：compile.js 应增加"目录 vs 实际标题"的覆盖率检查（当前仅检查结构/命名/链接/基线）
3. **验证维度应前置**：v1.0 验证报告的 4 维度未包含"文件完整性"维度，v2.0 新增的维度⓪应作为所有 A5 验证的标配

## 缺陷 #2: A2 问题清单覆盖率仅 4.7%

| 属性 | 值 |
|---|---|
| **缺陷编号** | DEF-EDU-2026-002 |
| **发现阶段** | A5 需求验证（维度② 问题清单比对） |
| **严重级别** | Major（影响 SRS 可信度） |
| **发现日期** | 2026-06-24 |
| **修复日期** | 2026-06-24（部分修复） |
| **当前状态** | ⚠️ 部分修复（覆盖率从 4.7% → ~55%，仍有 12 条 OPEN） |

### 2.1 缺陷描述

A2 需求分析智能体产出了 **64 条**问题（Academic Affairs 18 + Course Consultant 7 + Finance 10 + Principal 8 + System Admin 16 + 跨涉众冲突 5）。

SRS-v1.1 仅有效关闭了其中 **3 条 (4.7%)**，9 条有 [TBD] 标记，高达 **52 条 (81.3%)** 完全遗漏。

最严重的是 System Admin 模块的 16 条问题——因 §3.1.6 整章缺失，全部无 SRS 条目可追溯。

### 2.2 影响分析

| 影响维度 | 说明 |
|---|---|
| **SRS 可信度** | 仅有 4.7% 的问题关闭率意味着 SRS 未充分吸收需求分析的成果，涉众原话中的模糊/矛盾/不一致未得到处理 |
| **System Admin 涉众感知** | 16 条问题全部悬空，该涉众可能认为自己的需求"被 AI 忽略了" |
| **后续设计风险** | 未处理的问题将在架构设计/编码阶段暴露，修复成本呈指数增长 |

### 2.3 根因分析

| 环节 | 问题 |
|---|---|
| **A4 输入** | A4 生成 SRS 时可能未完整读取 `需求问题清单-v1.0.md`，或其 prompt 中未强调"逐条关闭 A2 问题" |
| **A4 token 限制** | 同缺陷 #1——SRS 的 token 配额被 FR 正文消耗，没有余量处理问题闭环 |
| **流程设计** | A4→A5 之间缺乏"A2 问题覆盖率自动检查"门禁 |

### 2.4 修复措施

1. 新增 §3.1.6 ADM 模块（FR-ADM-001~005），覆盖 System Admin 的 16 条问题
2. 为 Course Consultant / Finance / Principal 的剩余问题在对应 FR 中补充 [TBD] 标记
3. 所有 [TBD] 统一汇入 §4.1 清单，标注回退涉众和优先级

### 2.5 修复验证

| 指标 | 修复前 | 修复后 |
|---|---|---|
| 问题关闭率 | 3/64 (4.7%) | ~35/64 (~55%) |
| System Admin 覆盖 | 0/16 (0%) | 16/16 (100%，含 [TBD]) |
| 跨涉众冲突覆盖 | 1/5 (20%) | 5/5 (100%，均标记 [TBD]) |

### 2.6 经验教训

1. **A4 的 prompt 必须包含问题清单作为强制输入**，并要求"逐条在 SRS 中处理（关闭/标记 TBD/注明理由）"
2. **A5 验证前应有自动化门禁**：检查 `需求问题清单` 中每条问题是否在 SRS 中有对应引用
3. **分模块验证优于一次性全量验证**：可先验证覆盖率最高的模块，逐模块递进

## 缺陷 #3: 非功能需求/外部接口/数据需求三大章节完全缺失

| 属性 | 值 |
|---|---|
| **缺陷编号** | DEF-EDU-2026-003 |
| **发现阶段** | A5 需求验证（维度⑤⑥⑦） |
| **严重级别** | Blocker（3 个 Blocker 合并为一个缺陷族） |
| **发现日期** | 2026-06-24 |
| **修复日期** | 2026-06-24 |
| **当前状态** | ✅ 已修复 |

### 3.1 缺陷描述

SRS-v1.1 的 §3.2（外部接口需求）、§3.3（数据需求）、§3.4（非功能需求）三个 IEEE 830 强制章节**完全空白**——目录中有标题但正文为零。

这不是截断问题（截断发生在 §3.1.5），而是 A4 在生成到 §3.1.5 末尾时已经耗尽 token 配额，根本未开始生成这三个章节。

### 3.2 具体缺失内容

| 缺失章节 | 应有内容 | 当前状态(v1.1) |
|---|---|---|
| §3.2 外部接口 | 支付/短信/小程序/银行流水接口需求 | 完全空白 |
| §3.3 数据需求 | 逻辑数据模型(ER) + 6组状态枚举 | 完全空白 |
| §3.4 非功能需求 | 性能(并发/响应)/安全(加密/脱敏)/可用性(备份/灾备)/审计 | 完全空白 |

### 3.3 影响分析

**直接影响**：后续软件设计阶段（ADR/ASD/MDS/TLCD/OAS）将缺乏关键输入——
- 无 NFR → ADR-001 架构选型无性能/安全约束
- 无接口需求 → OAS 契约无定义依据
- 无数据模型 → MDS 模块划分缺少数据边界

### 3.4 修复措施

基于 `raw/notes/System Admin` 和 `raw/notes/Finance` 中的原始需求记录补全：

- **§3.2**: 5 条软件接口（微信/支付宝/短信/小程序推送/银行流水）+ UI/硬件/通信接口
- **§3.3**: 10 实体逻辑模型 + 6 组关键枚举（32 个状态值）
- **§3.4**: 15 条 NFR 覆盖 5 个子类（性能 4/安全 5/可用性 3/可维护性 3/审计 3）

### 3.5 修复验证

| 章节 | 补全条目 | compile.js | A5 重验证 |
|---|---|---|---|
| §3.2 | 5 条 IF-SW | ✅ | 维度⑥ 通过 |
| §3.3 | 10 实体 + 6 枚举 | ✅ | 维度⑦ 通过 |
| §3.4 | 15 条 NFR | ✅ | 维度⑤ 通过 |

### 3.6 经验教训

1. **IEEE 830 的"骨架先行"策略**：应先填充所有章节的骨架（标题 + 占位符），再逐章展开正文。这样即使 token 耗尽，至少所有章节都有占位标记
2. **token 预算分配**：SRS 生成应预估各部分 token 占比——功能需求占 50%、NFR 占 15%、接口占 10%、数据占 10%、附录占 15%

## 缺陷 #4: 跨涉众冲突未在 SRS 中处理

| 属性 | 值 |
|---|---|
| **缺陷编号** | DEF-EDU-2026-004 |
| **发现阶段** | A5 需求验证（维度⑧ 跨涉众冲突验证） |
| **严重级别** | Blocker（CFC-02/CFC-04）/ Major（其余 4 项） |
| **发现日期** | 2026-06-24 |
| **修复日期** | 2026-06-24 |
| **当前状态** | ⚠️ 部分修复（5/5 冲突已标记处理方案或 [TBD]） |

### 4.1 缺陷描述

A2 需求分析智能体检测出 **5 个跨涉众冲突**：

| 冲突 | 涉众 | 冲突主题 |
|---|---|---|
| CFC-01 | Academic Affairs vs System Admin | 名额锁定：即时自动化 vs 并发扩展性 |
| CFC-02 | Principal vs System Admin | 续费率定义：确定性报表 vs 公式未统一 |
| CFC-03 | Finance vs System Admin | 支付对账：实时同步 vs T+1 安全方案 |
| CFC-04 | Academic Affairs vs System Admin | 退费差价：期望自动计算 vs 规则未量化 |
| CFC-05 | Finance vs System Admin | 跨校区财务权限：统一管理 vs 严格隔离 |
| CFC-06 | System Admin 内部 | 数据隔离 vs 集团统计分析 |

SRS-v1.1 中，仅 CFC-01 在 FR-ACA-003 中有间接的 [TBD] 处理，其余 5 个冲突完全未被提及。

### 4.2 根因分析

**跨涉众冲突是 A2 的"跨文档"分析产物**——它不在任何单一涉众的原始记录中，而是 A2 综合分析 5 份记录后发现的。A4 生成 SRS 时按模块/涉众组织内容，天然倾向于"单涉众视角"，忽略了跨涉众的冲突维度。

### 4.3 修复措施

1. **CFC-01**: FR-ADM-001 覆盖多校区隔离 + TBD-15 覆盖并发/扩展性确认
2. **CFC-02**: FR-RPT-003 覆盖续费率分析 + TBD-13 覆盖公式确认
3. **CFC-03**: FR-ENR-005 + NFR-PERF-001 覆盖对账与响应；TBD 待确认时效性级别
4. **CFC-04**: FR-FIN-002 + FR-ADM-004 覆盖退款与阈值；转班差价规则补充 [TBD]
5. **CFC-05**: FR-ADM-001/002 + TBD-15 覆盖隔离与汇总；统一管理 vs 隔离矛盾待确认
6. **CFC-06**: FR-ADM-001 + TBD-15 覆盖技术方案选项

### 4.4 经验教训

1. **A4 prompt 应显式要求"处理跨涉众冲突"**：将 `需求问题清单` 中的冲突节作为强制输入
2. **跨涉众冲突应在 SRS 中有专属章节**：或在每个相关 FR 中引用，或在 §2.5 假设与依赖中专门讨论
3. **[TBD] 是可接受的中间态**：对于无法从现有素材推导的冲突解决，标记 [TBD] 并指定回退涉众比强行决策更合理

## 缺陷 #5: RTM 需求溯源矩阵缺失

| 属性 | 值 |
|---|---|
| **缺陷编号** | DEF-EDU-2026-005 |
| **发现阶段** | A5 需求验证（维度⑨ RTM 完整性验证） |
| **严重级别** | Blocker（期中考核必备交付物缺失） |
| **发现日期** | 2026-06-24 |
| **修复日期** | 2026-06-24 |
| **当前状态** | ✅ 已修复 |

### 5.1 缺陷描述

SRS-v1.1 目录中承诺了 §4.2 需求溯源矩阵（RTM），但实际内容完全空白。

作战计划 §2.6 要求 RTM 包含全部 FR 条目与原始需求记录、UML 模型的追溯关系。期中考核交付物清单第 8 项为"需求基线文档（A6 正式 SRS + RTM 溯源矩阵）"。

### 5.2 影响分析

| 影响 | 说明 |
|---|---|
| **期中考核合规** | 缺少 RTM 将导致期中第 8 项交付物不完整 |
| **需求变更管理** | 无 RTM 将无法在 CIA 阶段评估变更影响范围 |
| **可追溯性审计** | 无法证明每条 FR 源自哪位涉众的哪一轮对话 |
| **作战计划对齐** | §2.6 要求 RTM 为基线必备项，"ID 终身唯一、不复用" |

### 5.3 根因分析

同缺陷 #1——SRS 在 §3.1.5 处截断，A4 根本没有机会生成 §4 附录的任何内容。

### 5.4 修复措施

1. 逐条追溯全部 29 条 FR 的来源：原始需求记录（涉众 + 轮次）+ A2 问题清单 + UML 模型
2. 以 5 列表格形式（需求ID / 描述摘要 / 来源涉众 / 原始记录 / 问题清单 / UML / 状态）输出
3. 标注每条 FR 的状态：✅ 已确认 / ⚠️ 含 TBD

### 5.5 修复验证

| 验证项 | 结果 |
|---|---|
| FR 总数 | 29 条 |
| 已确认 | 22 条 (75.9%) |
| 含 TBD | 7 条 (24.1%) |
| 涉众覆盖率 | 5/5 (100%) |
| UML 覆盖率 | 4/4 (100%) |
| A5 重验证 | 维度⑨ 通过，RTM-001 关闭 |

### 5.6 经验教训

1. **RTM 不应是"附录"——它应是 SRS 的核心组件**：可考虑将 RTM 放在 §3 具体需求之后、§4 附录之前
2. **RTM 应在 FR 编写时同步建立**：每个 FR 写完立即填写溯源行，而非全部 FR 写完后再回头追溯
3. **compile.js 应增加 RTM 完整性检查**：验证 RTM 中的 FR 编号是否覆盖了 SRS 正文中所有 `FR-` 条目
