## 1. 决策背景

基于 SRS-v1.2（771行，29 FR + 15 NFR + 5 IF-SW）进行五维度架构选型。技术栈已预决策为 **Java + Spring**（作战计划 §8 决策 1），本 ADR 在此约束下进行细化选型。

### 关键驱动需求

| 需求编号 | 需求 | 架构影响维度 |
|---|---|---|
| NFR-PERF-001 | 100并发，核心交易≤3秒 | 分层 / 部署 |
| NFR-AVL-001 | 可用性≥99.5% | 部署 |
| NFR-AVL-002 | RPO≈5min, RTO≈4h | 部署 / 数据 |
| NFR-SEC-001 | AES-256加密，脱敏展示 | 安全 |
| NFR-SEC-004 | HTTPS双向校验，支付验签 | 安全 / 集成 |
| NFR-AUD-002 | 操作前后值快照，完整审计链 | 数据 |
| FR-ADM-001 | 多校区数据隔离 + 集团汇总 | 数据 / 部署 |
| IF-SW-001~005 | 微信/支付宝/短信/小程序/银行流水接口 | 集成 |
| FR-ENR-004 | 支付即记录，财务一键对账 | 集成 |

## 2. 五维度选型分析

### 维度 1: 部署架构

| 方案 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| **A: 单机单体** | 单个 Spring Boot JAR，内嵌 Tomcat，单 MySQL 实例 | 开发简单、部署快 | 无法满足 99.5% 可用性、无灾备 |
| **B: 云原生微服务** | Kubernetes + Spring Cloud，多服务独立部署 | 弹性伸缩、独立升级 | 团队规模不匹配（单人项目）、运维复杂度高 |
| **C: 模块化单体 + 主备** | Spring Boot 单体应用（按功能分包），MySQL 主从 + 异地备份，Nginx 反向代理 | 满足可用性要求、运维简单、支持后续拆分 | 单点瓶颈（可通过垂直扩展缓解） |

**✅ 决策: 方案 C — 模块化单体 + 主备**

| 决策依据 | 对应 SRS 需求 |
|---|---|
| 项目团队为单人，微服务运维 overhead 超出收益 | §2.4 技术约束 |
| 模块化分包（按 CRM/ENR/ACA/FIN/RPT/ADM 6 模块）支持未来向微服务演进 | — |
| MySQL 主从复制 + 异地每日全量备份 → RPO≈5min, RTO≈4h | NFR-AVL-002 |
| Nginx 反向代理 + 静态资源分离 → 支持 100 并发 | NFR-PERF-001 |

---

### 维度 2: 分层架构

| 方案 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| **A: 三层（Controller→Service→DAO）** | 传统 Spring MVC 三层 | 简单 | 缺乏领域抽象，Service 层易膨胀 |
| **B: 四层（Controller→Service→Repository→Model）** | Controller→Service→Repository→Model，作战计划 §8 既定 | 职责清晰、Repository 解耦数据访问 | 略增文件数 |
| **C: DDD 六边形架构** | 领域驱动设计，Ports & Adapters | 高度解耦 | 学习曲线高、小项目过度设计 |

**✅ 决策: 方案 B — Controller→Service→Repository→Model 四层**（作战计划既定）

| 层级 | 职责 | 约束 |
|---|---|---|
| **Controller** | HTTP 请求接入、参数校验、响应封装 | 不含业务逻辑，不直接访问数据库 |
| **Service** | 业务逻辑编排、事务管理、权限校验 | 不含 SQL，通过 Repository 访问数据 |
| **Repository** | 数据访问抽象（JPA/MyBatis）、查询封装 | 不含业务逻辑 |
| **Model** | 实体定义、DTO/VO、枚举、异常类 | 纯数据结构 |

```
┌─────────────────────────────────────┐
│           Controller 层              │  ← HTTP/REST 接入
├─────────────────────────────────────┤
│            Service 层                │  ← 业务逻辑 + 事务
├─────────────────────────────────────┤
│          Repository 层               │  ← 数据访问抽象
├─────────────────────────────────────┤
│            Model 层                  │  ← Entity/DTO/VO/Enum
└─────────────────────────────────────┘
        跨层横切: Security / Audit / Log
```

### 维度 3: 数据架构

| 方案 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| **A: 单库单 Schema** | 所有校区数据存入同一 Schema，通过 `campus_id` 字段隔离 | 查询简单、集团汇总容易 | 数据隔离弱（依赖应用层 WHERE 过滤），误操作风险 |
| **B: 单库多 Schema** | 每个校区独立 Schema，集团汇总通过跨 Schema 视图 | 逻辑隔离强 | MySQL 跨 Schema 查询性能差 |
| **C: 单库 + Row-Level Security** | 单 Schema + 应用层强制 `campus_id` 过滤 + 审计日志验证 | 兼顾查询性能与隔离 | 依赖应用层严格实现 |

**✅ 决策: 方案 C — 单库 + 应用层 Row-Level Security**

| 决策依据 | 对应 SRS 需求 |
|---|---|
| 所有查询通过 Repository 基类自动注入 `campus_id` WHERE 条件 | FR-ADM-001 (数据隔离) |
| 集团汇总通过只读视图（`campus_id IS NOT NULL` 聚合） | FR-ADM-001 (集团汇总) |
| 操作日志记录每次数据访问的 `campus_id`，定期审计验证 | NFR-AUD-002 (操作追溯) |
| [TBD] 未来集团数据量大时可引入数仓副本（TBD-15） | FR-ADM-001 [TBD] |

### 数据访问技术选型

| 技术 | 用途 | 理由 |
|---|---|---|
| **Spring Data JPA** | 常规 CRUD 操作 | 标准、成熟、与 Spring 生态深度集成 |
| **MyBatis** | 复杂报表查询（FR-RPT-001~005） | 灵活 SQL 控制，适合多表关联和聚合 |
| **Flyway** | 数据库版本迁移管理 | 版本化 DDL，支持 CI/CD 回滚 |
| **Redis** | 会话管理 + 支付幂等性令牌 + 排课冲突检测缓存 | NFR-PERF-001 (≤3s), NFR-SEC-005 (防重复提交) |

---

### 维度 4: 安全架构

| 关注点 | 决策 | 对应 SRS 需求 |
|---|---|---|
| **身份认证** | Spring Security + JWT（后台管理端）；微信 OAuth 授权（小程序端） | NFR-SEC-002 |
| **权限控制** | RBAC 五角色（顾问/教务/财务/校长/管理员）+ 自定义角色，Spring Security `@PreAuthorize` 方法级注解 | FR-ADM-002 |
| **数据加密** | AES-256 加密敏感字段（姓名/手机号/身份证），使用 Spring Crypto `Encryptors` | NFR-SEC-001 |
| **传输安全** | 全站 HTTPS，支付接口证书双向校验 | NFR-SEC-004 |
| **支付安全** | 支付回调验签（微信 API v3 平台证书、支付宝公钥验证），支付幂等性令牌（Redis 10 分钟过期） | IF-SW-001/002, NFR-SEC-005 |
| **数据脱敏** | 查询列表返回脱敏手机号（中间 4 位 `*`），使用 Jackson 自定义序列化器 | NFR-SEC-001 |
| **防越权** | Repository 基类强制注入 `campus_id` 过滤 + AOP 切面校验数据范围 | NFR-SEC-003 |
| **操作审计** | AOP 切面拦截 `@Auditable` 注解方法，记录操作前后值快照到 `audit_log` 表 | NFR-AUD-002, FR-ADM-003 |

---

### 维度 5: 集成架构

| 外部系统 | 集成方式 | 关键约束 |
|---|---|---|
| **微信支付** | Spring Boot HTTP Client + 微信商户 API v3 SDK | 异步通知验签、支付幂等性、退款原路返回 |
| **支付宝支付** | Spring Boot HTTP Client + 支付宝 SDK | 同上 |
| **短信网关** | HTTP/HTTPS 模板短信 API，异步发送（消息队列解耦） | 防骚扰规则（TBD-20）：每日上限/免打扰/黑名单 |
| **微信小程序推送** | 小程序订阅消息 API | 需家长主动订阅 |
| **银行流水导入** | 文件导入（CSV/Excel，Apache POI 解析），自动匹配订单号+金额 | 降级匹配逻辑（FR-ENR-005） |
| **内部模块间** | RESTful API over HTTPS，JSON，JWT | — |

**消息队列选型**: 使用 **Spring Event + @Async**（轻量级，适合单体应用）处理短信发送、审计日志写入等非关键路径操作。未来如需高可靠异步，可迁移至 RabbitMQ。

## 4. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| 单体应用随模块增加而膨胀 | 中 | 中 | 6 模块分包清晰，边界明确；Adapters 层隔离外部依赖，为未来拆分预留接口 |
| 应用层 Row-Level Security 实现遗漏 | 低 | 高 | Repository 基类强制注入 + AOP 校验 + 定期审计报告 |
| Redis 单点故障 | 低 | 中 | Redis Sentinel 主从 + 自动故障转移；核心业务（支付）有数据库兜底 |
| 支付网关不可用 | 低 | 高 | 降级方案（TBD-22）：标记为待补录，T+1 人工对账 |
