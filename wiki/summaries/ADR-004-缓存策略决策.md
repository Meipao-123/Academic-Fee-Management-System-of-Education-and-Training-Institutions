# ADR-004: 缓存策略决策

**项目**: 培训机构教务收费管理系统 (EDU)
**ADR 编号**: ADR-004
**决策日期**: 2026-06-24
**决策者**: A5-需求验证智能体
**状态**: ✅ 已决策
**关联文档**: ASD-EDU-V1 (§4.1 内部通信) / ADR-001 (§2 维度 5) / SRS-v1.2 (§3.4.1 性能需求 NFR-PERF-001/002)

---

## 1. 决策背景

ASD 已确定 Redis 作为缓存中间件，但以下细节尚未决策：

| 待决策项 | 背景 |
|---|---|
| Redis 数据结构选择 | String vs Hash vs List vs Set vs SortedSet 各场景适用性 |
| TTL 策略 | 不同业务场景的缓存过期时间 |
| 缓存更新/失效模式 | Cache-Aside vs Write-Through vs Write-Behind |
| 缓存穿透/击穿/雪崩防护 | 空值缓存、互斥锁、随机 TTL |
| 哪些数据需要缓存 | 全量数据 vs 热点数据 |

---

## 2. 决策分析

### 决策 1: 缓存更新模式

| 方案 | 描述 | 适用场景 |
|---|---|---|
| A: Cache-Aside | 应用手动管理缓存：读→缓存未命中→查库→写缓存；写→更新库→失效缓存 | 读多写少，通用场景 |
| B: Write-Through | 写操作同步更新缓存和数据库 | 数据一致性要求极高 |
| C: Write-Behind | 写操作先写缓存，异步批量写库 | 写吞吐量极高，容忍短暂不一致 |

**✅ 决策: 方案 A — Cache-Aside（默认），支付幂等令牌场景使用 Write-Through**

| 场景 | 模式 | 理由 |
|---|---|---|
| 学员信息、课程列表、排课表 | Cache-Aside | 读多写少，缓存未命中时查库 |
| 支付幂等令牌 | Write-Through | 令牌必须在支付前写入 Redis，防止并发重复支付 |
| 系统配置（FR-ADM-004） | Cache-Aside + TTL 5min | 配置变更频率低，短 TTL 自然刷新 |

### 决策 2: Redis 数据结构分配

| 业务场景 | 数据结构 | Key Pattern | Value | TTL |
|---|---|---|---|---|
| **学员基本信息** | Hash | `student:{id}` | `{name, phone, campus_id, status}` | 30 min |
| **课程列表（按校区）** | Hash | `courses:campus:{id}` | `{courseId: courseName}` | 1 hour |
| **班级名额（实时）** | String | `class:quota:{classId}` | 剩余名额数（原子 DECR） | 永久（排课结束时 DEL） |
| **支付幂等令牌** | String | `payment:idempotent:{orderNo}` | `"1"` | 10 min |
| **排课冲突检测（临时）** | Sorted Set | `schedule:conflict:{teacherId}:{date}` | 时间段 score=t_start, member=t_end | 排课完成后 DEL |
| **调课超时释放队列** | Sorted Set | `reschedule:timeout` | score=截止时间戳, member=调课ID | 定时任务消费 |
| **系统配置** | Hash | `sys:config` | `{key: value}` | 5 min |
| **会话 (JWT Refresh)** | String | `refresh:{userId}:{tokenId}` | token 哈希 | 7 days |
| **接口限流计数器** | String | `ratelimit:{ip}:{api}:{minute}` | 请求计数（INCR + EXPIRE） | 1 min |
| **校长驾驶舱缓存** | String (JSON) | `dashboard:campus:{id}` | 预计算 JSON | 5 min |

### 决策 3: TTL 分层策略

| 层级 | TTL 范围 | 典型场景 |
|---|---|---|
| **热数据** | 1~10 min | 支付幂等令牌、排课冲突、接口限流、驾驶舱预计算 |
| **温数据** | 30 min ~ 1 hour | 学员信息、课程列表、系统配置 |
| **冷数据** | 1 day ~ 7 days | JWT Refresh Token、历史报表快照 |
| **永久（手动管理）** | 无 TTL | 班级名额（排课时 SET，结课时 DEL） |

### 决策 4: 缓存防护策略

| 风险 | 场景 | 防护措施 |
|---|---|---|
| **缓存穿透** | 查询不存在的 key（如 student:-1） | **空值缓存**：查询不存在的数据也缓存空标记（`{"__null": true}`），TTL 1 min |
| **缓存击穿** | 热点 key 过期瞬间大量请求打到 DB | **互斥锁**：缓存未命中时，用 `SETNX lock:key` 获取锁，只有获锁线程查库写缓存，其余线程等待后重读缓存 |
| **缓存雪崩** | 大量 key 同时过期 | **随机 TTL**：在基础 TTL 上叠加 ±20% 随机偏移（如 30min ± 6min）；**永不过期 + 异步刷新**：核心数据（如课程列表）设置逻辑过期时间，后台定时刷新 |
| **Big Key** | 单个 key 的 value 过大（>10KB） | **拆分**：`student:{id}` 用 Hash 而非 String JSON；禁止缓存整个数据表 |

### 决策 5: 缓存一致性权衡

| 场景 | 一致性要求 | 策略 |
|---|---|---|
| 支付幂等令牌 | 强一致 | Write-Through，Redis 写成功才允许后续流程 |
| 班级名额 | 强一致 | Redis DECR 原子操作 + 定时（每分钟）与 DB 对账 |
| 学员信息 | 最终一致 | Cache-Aside，更新 DB 后 `DEL student:{id}` |
| 驾驶舱预计算 | 最终一致 | 定时任务（每 5min）全量刷新，容忍短暂数据延迟 |
| 系统配置 | 最终一致 | 配置修改时 `DEL sys:config`，下次读取时自动重建 |

---

## 3. 决策影响

| 影响维度 | 说明 |
|---|---|
| **性能** | 缓存命中率目标 >85%（学员查询/课程列表），驾驶舱首页加载从 >5s → <2s（NFR-PERF-002） |
| **内存** | 预估 Redis 内存占用：热数据 <100MB（500学员×200B + 100课程×100B + 临时期限数据） |
| **运维** | Redis Sentinel 哨兵模式保证高可用，单节点故障自动切换 |
| **编码规范** | 所有缓存操作通过 `core.cache.CacheService` 统一封装，禁止直接使用 `RedisTemplate` |

## 4. 缓存操作代码规范

```java
// ✅ 正确：通过 CacheService 统一接口
@Autowired
private CacheService cacheService;

public StudentDTO getStudent(Long id) {
    return cacheService.getOrLoad(
        "student:" + id,
        StudentDTO.class,
        () -> studentRepository.findById(id).map(StudentDTO::from),
        Duration.ofMinutes(30).plusMillis(ThreadLocalRandom.current().nextLong(360_000)) // ±20%
    );
}

// ❌ 错误：直接操作 RedisTemplate
redisTemplate.opsForValue().set("student:" + id, student);
```
