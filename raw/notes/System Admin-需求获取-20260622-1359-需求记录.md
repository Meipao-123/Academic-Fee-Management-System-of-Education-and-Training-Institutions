### 1. 当前实际流程（无自动化校验）

在现有Excel+线下协作模式下，**操作流程如下**：

- **数据入口**：教务人员根据学员的报名记录、课程A和B的单价、剩余课时等，在Excel中手工计算差价。例如：  
  `应退金额 = 课程A实付金额 - (课程A已消耗课时 × 课程A单价)`
  `应补金额 = 课程B总价 - (课程B已消耗课时 × 课程B单价)`（通常课程B尚未开始，此项为0）
  `实际退费 = 应退金额 - 应补金额`

- **提交审批**：教务填写纸质或简易电子《退费/转班申请单》，附上手工计算过程及备注（如“教材已领-扣50元”），提交给校区主管。

- **主管审核**：主管通常凭经验判断金额是否“合理”。例如对比同期类似转班案例，或电话询问教务“这个差价是怎么算出来的”。若无疑问则签字，转入财务。

- **财务复核**：财务人员需重新核对计算逻辑，并检查扣费项目（如已发教材、体验课消耗）。由于缺乏系统记录，常需与教务多次电话沟通确认。

### 2. “半自动化”下的典型异常与纠纷场景

| 异常类型 | 典型案例 | 后果 |
|---------|---------|------|
| **差价计算错误** | 教务将“课程A已消耗课时（共40节）”算成24节，实际是22节，导致多退2节课的钱（约300元）。主管审批时未发现，财务发现后需反向调整，引发家长投诉“退费金额前后不一致” | 多退少补流程复杂；家长信任受损 |
| **扣费项目遗漏或重复** | 课程A曾赠送过教材（价值80元），教务在计算退费时未扣除；而课程B也有教材费（含在总价中），家长要求只退差额时，系统无法自动识别哪些属于“已发生成本”，导致矛盾 | 校区承担额外成本或需线下解释 |
| **流程状态模糊** | 转班申请已提交，但主管出差未批复；家长同时已在课程B上课（实际完成了转班），但财务端显示“退费待批”，最终需人工做“事后补录”，造成账务日期错位 | 财务报表失真；审计风险 |
| **多校区数据冲突** | A校区学员要求转至B校区，但两个校区对“已消耗课时”的认定标准不同（A按“签到次数”，B按“总课时-剩余课时”），导致计算差异，两地教务相互推诿 | 跨校区协调成本高，影响用户体验 |

### 问题一：关于“已消耗课时”的认定标准

**1. 统一标准的选择：强制统一为“学员实际签到记录”作为唯一权威依据。**

*   **理由：** 我们的核心目标是实现自动化、可追溯、不可篡改的跨校区业务流转。“总课时−剩余课时”这个口径受限于“剩余课时”定义的不统一（例如，剩余课时是否包含已过期未上的课时？是否包含赠送课时？）。而“签到记录”是学员实际上课的客观事实，只要系统采集数据正确，它就是最可靠的。
*   **实施方案：**
    *   **系统配置：** 在【基础管理】-【系统配置】中，我会设置一个全局开关：“课时消耗标准”，强制所有校区必须选择“签到记录”。未来新校区上线时，这个默认值不可修改，从源头解决数据异构问题。
    *   **折算映射（备选方案）：** 否决。允许“折算映射表”会引入数据转换误差和维护复杂度。对于A、B校区的历史数据差异，我们将在历史数据迁移阶段，统一将所有校区的历史“已消耗课时”按“学员实际签到记录”进行清洗和补录（需要各校区提供历史签到台账）。迁移后，系统将只承认未来的签到数据。
    *   **跨校区转班：** 自动以学员在A校区的签到记录作为“已消耗课时”，该数据写入数据库，不可修改。B校区系统读取该数据后，自动完成差价计算。

**2. 签到数据的权威性与兜底机制：**

*   **采集渠道：** 我们将在新系统中要求所有校区**必须采用系统内置的“电子签到”功能**（支持校区主管通过教务端、或者学员在小程序端扫码/人脸识别签到）。纸质签到表或手工录入将不作为系统认可的合法消耗记录。这是为了满足**安全性**和**可靠性**指标。
*   **兜底处理：**
    *   **课时消耗申诉通道：** 设计一个“课时异常消耗申诉”功能。该功能**仅限校区主管**发起，需要填写申诉原因（如学员请假未标记、系统Bug导致未签到等），并上传线下签字确认的纸质签到表照片作为附件。
    *   **审批流程：** 该申诉单会自动流转至**教务系统管理员（我所在的岗位）** 进行审核。审核通过后，我会在后台手动修改该学员特定课次的签到状态（修改行为会生成操作日志，确保可追溯）。**不允许校区主管自行修改签到数据**，避免滥用。

### 问题二：扣费项目的基础字典与扣减逻辑

**1. 字典的维护权限：实行“总部统管、分校申请”的双重控制。**

*   **总部统一配置：** 教材、体验课、赠品等基础扣费项目的**名称、扣减标准**由总部统一在【基础管理】-【系统配置】中维护，形成“核心字典”。这些是跨校区转班时系统必须能够识别的项目，不可由校区自行新增或修改。
*   **校区自维护权限（受限）：** 允许各校区在“核心字典”基础上，**申请**新增“子项”。例如，校区A特有的“XX学科材料包”，必须向总部提交申请，总部审核通过后，为其分配一个**唯一的、全局的、可关联的编码**，而不是仅靠名称。这样，在转班时，若B校区没有该子项，系统会提示“此子项无对应标准，无法自动计算”，需要人工干预（由校区主管在转班申请时手工添加一个折价金额，但该金额需在总部配置的限额内）。
*   **设计原则：** 基础字典稳定，不可随意增删；子项字典可扩展，但需编码一致，确保数据可关联。

**2. 扣减触发条件：自动关联 + 人工勾选，并明确人工干预边界。**

*   **自动化关联：** 对于**出库记录可追踪**的扣费项目（如印刷教材、教具），系统应当与已有的“教材库房管理”或“领用登记”模块联动。当系统识别到该学员的报名记录中关联了某个教材包，且在系统内有一笔“领用出库”记录时，该教材对应的扣费项目**自动挂起**。
*   **人工勾选：** 对于**无法关联出库记录**的项目（如体验课时、一次性领取的赠品、教材已被涂写无法退还），流程设计如下：
    *   当校区主管发起“转班申请”时，系统会弹出一个**“已发生成本确认清单”**，里面会列出该学员在报名时关联的所有可能扣费项目。
    *   教务主管**必须手动勾选**那些**实际已发放或已使用**的项目（例如，勾选“体验课X节”、“教辅材料已领取”）。
*   **人工干预边界（关于全额扣除和折扣）：**
    *   **全额扣除：** 对于“一次性消耗品”（如已用体验课）或“无法回收的赠品”（如已拆封的教具），**系统强制执行全价扣除**，校区主管无权限修改金额。这是为了标准化。
    *   **酌情折扣：** 针对“教材已涂写但部分可用”这类特殊情况，系统允许校区主管在勾选该项目后，**手动输入一个扣减折扣比例（例如80%）**。但这个功能有**严格限制**：
        *   **比例上限：** 总部设定的最高比例下限（例如最低只能折扣到原价的70%），防止为了拉拢生源而故意低评。

### 一、关于“非标准消耗”的数据标记与影响

您提出的“赠送课时”和“请假/停课”场景非常关键，直接影响到“已签到课时”和“已发生不可退扣费”的统计准确性。我们的系统设计方案如下：

1.  **【赠送课时】处理方案**
    *   **标记方式**：在课程订单或学员课程明细中，为赠送课时设置一个独立的类型字段（例如：`课时类型= “赠送”`）。其对应的签到记录会存在，但系统会为每次签到生成一个**零单价**的计费行（`实付单价=0`）。
    *   **对公式的影响**：
        *   `已签到课时`：**会计入**。例如课程总课时20节，赠送2节，学员签到了15节（其中包含1节赠送课），则已签到课时=15。
        *   `(A实付总额/A总课时)`：**不包含赠送课时**。分母`A总课时`应理解为**总付费课时**（即18节），而非总课时（20节）。赠送课时不参与实付单价的计算。
        *   **结论**：赠送课时只影响“已消耗课时”的计数，但不产生“已消耗价值”。在计算`原课程A剩余价值`时，系统会自动剔除赠送课时的价值贡献。**教务核对数据时，可以直观看到签到记录中有“赠送”标签，但无需手动计算其价值。**

2.  **【请假/停课】处理方案**
    *   **标记方式**：
        *   **请假（短期）**：在考勤模块中，请假会生成一条**“请假记录”**，系统会将其状态标记为`不计消耗`。它不影响`已签到课时`的统计（已签到课时只统计实际出勤的课时）。
        *   **停课（长期）**：会在学员课程状态上打上`停课中`的标记，该课程所有后续的课时计划（如排班）将被冻结，不再消耗。已消耗的课时以停课前的实际签到记录为准。
    *   **对公式的影响**：
        *   请假/停课**不产生**已签到记录，因此`已签到课时`完全不受影响。
        *   但需要注意：如果请假发生在“已缴费但未上课”的时段，它不属于`已发生不可退扣费`，这部分课时（请假但未消耗）理应计入剩余价值。我们会将“请假未消耗”的课时与“未签到但可用”的课时在逻辑上统一视为“可消耗课时”，但**不计入已签到课时**。
        *   **结论**：请假不影响`已签到课时`和`已发生不可退扣费`的计算，因此不会改变剩余价值。系统会自动处理这部分“冻结”的课时状态。

### 二、关于审批流设计的确认

    *   **财务复核触发条件**：财务复核**不是默认流程**。它仅在以下两种情况下被自动触发：
        1.  **人工调整导致偏差超过10%阈值**（如您所述）。
        2.  **补费或退费金额超过校区主管的审批权限**（该权限由校区自行定义，并配置在`系统配置`中，例如单笔超2000元需财务复核）。**其他情况，校区主管审批后，系统自动完成后续操作（如生成新课程订单、调整课时）。**

2.  **审批界面关键信息（数据看板设计）**
    *   **校区主管视角**：核心是快速判断“该笔转班是否合理、有无风险”。

        *   **课程信息**：原课程名称、总课时/已签到课时/剩余课时、实付总额、剩余价值（系统自动计算）。

        *   **差值计算**：`补费/退费金额 = 新课程总价 − 原课程剩余价值`。系统自动计算并高亮显示该数值。
        *   **阈值告警**：`偏差率 = |人工调整金额| / 原课程剩余价值`。如果触发了10%阈值，会弹出红色告⚠️并标明“需财务复核”。
        *   **出勤记录摘要**：最近30天的签到率，帮助主管判断出勤习惯（如长期请假可能影响剩余价值准确性）。
    *   **财务视角**：核心是验证金额计算准确性、支付安全性。
        *   **支付流水历史**：原课程所有缴费记录（微信/支付宝支付单号、金额、时间）。

**问题一：关于临时授权续期**

**结论：支持在原授权到期前，由集团管理员直接延长。** 这是我们权衡了操作便捷性与安全审计后决定的方案。

**理由：**
1.  **避免流程断层**：若要求主管重新提交完整申请，意味着原授权链中已被“批准”但尚未执行的审批记录，在逻辑上会中断。例如，主管A申请授权给教务B处理“学员X的退费”，教务B处理到一半（比如已生成第一级审批单，等待财务审核），此时授权到期。若重走流程，新授权无法与旧操作无缝衔接，会造成业务混乱。直接延长则可保持审计链的完整性。
2.  **操作效率**：灵活的业务场景下（如突发跨天旅游），让主管重新填写全部申请理由和范围，增加不必要的管理成本。

**具体实现：**
-   **操作方式**：集团管理员在“临时授权管理”列表中，对处于“生效中”或“即将到期（剩余<4小时）”的授权条目，点击“延长”按钮。
-   **延长机制**：
    -   延长操作本身作为一条独立的审计日志记录（操作人、时间、原授权ID、延长后到期时间）。
    -   延长的总时长**有上限**：**累计不超过72小时**。即：初始24小时 + 续期次数/时长总和不超过72小时。
    -   **到期前提醒机制升级**：在授权到期前6小时、1小时，不仅提醒被授权人，也会通过系统消息、企业微信/钉钉通知**申请主管**，提醒其关注是否需要续期。

**问题二：关于临时授权范围锁定**

**结论：采用“预配置条件 + 动态细粒度限制”的双层锁定模式。**

**具体实现：**

1.  **第一层：预配置条件（由集团管理员在系统后台配置）**
    -   这是硬性过滤规则，由集团管理员在“授权模板”或“授权配置”中预定义。例如：
        -   `金额条件`：退费金额 ≤ 1000元。
        -   `单据类型`：仅限“退费审批单”。
        -   `校区范围`：限定为“XX校区”。
    -   校区主管在发起临时授权申请时，只能从管理员配置好的条件中选择。**不允许主管自行创建或修改这些条件。** 这样就杜绝了主管“勾选全校所有班级”的可能性，因为条件本身已被限定。

2.  **第二层：动态细粒度（由申请主管在申请时指定）**
    -   在第一层条件框定的范围内，主管可以进一步细化，以应对真实业务场景。例如：
        -   **指定学员**：输入学员姓名或学号，系统自动校验该学员是否属于XX校区。
        -   **指定班级**：从预置列表中选择“XX校区-XX班级”。
        -   **指定单据ID**：输入具体的退费申请单号。
    -   **防扩大机制**：
        -   **下拉/搜索限制**：主管在指定范围时，系统仅展示该校区/该类型/该金额条件下的数据。例如，主管想选择“全校所有班级”，但下拉菜单中只有其所在校区的班级，无法选择其他校区班级。
        -   **变更历史记录**：主管每次新增、修改、删除指定范围的操作，都会记录在审计日志中，便于事后追溯。
        -   **强制最小粒度原则**：系统会在界面上提示“建议指定到具体学员/单据，以避免不必要的数据暴露”。如果主管勾选范围过大（例如勾选了整个班级30人），系统会弹出确认框：“您即将授权处理本班级所有学员的退费，是否有必要？建议指定具体学员。” 但这并非强制阻断，只是提醒，最终由主管自负责任。

#### 3. 强依赖关系

您指出的依赖关系非常精准。从系统管理员视角，依赖关系列表如下：

1.  **【用户权限管理】 → 【退费审批流】 (强依赖)**

2.  **【备份策略】 → 【支付接口合规】 & 【历史数据导入】 (间接依赖，但需并行设计)**
    *   **原因**：备份策略的正确性直接影响数据传输和存储的合规性。如果备份方案不合理（如全量备份未加密），导致支付接口的加密数据被明文备份，则违反了数据安全法规。**在设计支付接口时，必须同步考虑审计日志、财务数据、学员个人数据的备份加密方式**，确保加密的完整性。同样，历史数据导入后，也需要纳入备份策略中。所以，虽然不构成绝对的时间先后，但**必须并行设计并确保逻辑一致性**。

3.  **【多校区数据隔离】 → 【用户权限管理】 (并行依赖)**
    *   **原因**：用户权限是定义“谁可以访问哪些数据”的，而数据隔离是定义“数据是如何被分隔的”。二者必须协同工作。例如，一个拥有“校区主管”角色的用户，只能操作自己校区内的数据。**数据隔离的设计（如数据库字段`campus_id`）必须先于权限校验逻辑的编写**，否则权限校验无法进行。因此，**数据隔离与用户权限管理应同步规划，但数据隔离的存储方案需先行确认**。

4.  **【系统配置】 → 【历史数据导入】 (弱依赖)**
    *   **原因**：在导入历史数据前，系统的基础参数（如课程分类编码、校区代码）应该已经配置完成，导入模板才能与这些参数映射。但可以先导入原始数据，再在后续进行映射，因此依赖不强，可并行处理。

**总结**：**强依赖链条是：多校区数据隔离方案 → 用户权限模型设计 → 退费审批流与数据访问控制**。而**备份策略与支付接口设计、历史数据导入方案需要并行协同**，不能互相割裂。

   ```
   差价 = 原剩余价值 − 新应收金额（正退负补）
   ```
3.  **审批流**：
   - **标准链**：所有转班申请必须校区主管审批。跨校区/补费标记“高关注”。
   - **财务复核触发**：人工调整额偏差率>全局参数`10%`，或补/退费金额>校区主管授权限额。
   - **状态管理**：审批与支付解耦，退款异步重试（指数退避，5次失败后通知财务）。
4.  **边界规则**：
   - 同学员日仅一次待审批转班。
   - 二次退费延伸同一实付单价和资金链。
   - 阶梯促销暂不细化，后续迭代。
